# Repository Guidelines

## Project Structure & Module Organization
1. Root: `README.md` (полный список), `README.short.md` (краткий приоритетный список);
2. Optional (при появлении): `assets/` для изображений, `drafts/` для черновиков;
3. Поддерживайте согласованность: краткая версия отражает ключевые пункты из полной.

## Build, Test, and Development Commands
1. Build: не требуется (Markdown-only);
2. Preview: используйте предпросмотр Markdown в редакторе;
3. Lint (опционально): `npx markdownlint .` — проверка форматирования Markdown;
4. Link check (опционально): `npx markdown-link-check README.md` — проверка ссылок.

## Coding Style & Naming Conventions
1. Язык: русский; английский — только там, где явно требуется (например, раздел “Architecture and System Evolution”);
2. Тон: профессиональный, без эмодзи;
3. Заголовки: Markdown `#`, `##`, `###`; заголовки обычным регистром (не Title Case);
4. Списки: нумерованные (1., 2., …); формулировки короткие и результат-ориентированные;
5. Наименования технологий: используйте каноническое написание (Kubernetes, GitLab, GitHub, Envoy, etcd).

## Правила списков
1. Делать списки нумерованными;
2. В конце каждого пункта в списке должна быть точка с запятой, кроме последнего пункта в данном списке, он должен заканчиваться точкой.

## Testing Guidelines
1. Согласованность ролей в `README.md`: сначала tech lead, затем backend developer, затем прочее;
2. `README.short.md` — только наиболее значимые пункты, без ссылок;
3. Орфография: проверяйте RU/EN; избегайте смешения транслитераций;
4. Ссылки (в полной версии): без UTM; проверяйте доступность.

## Commit & Pull Request Guidelines
1. Сообщения коммитов: повелительное наклонение, краткая суть (≤72 симв.), при необходимости — тело с деталями;
2. Изменения атомарны: одна логическая правка — один коммит;
3. PR: чёткое описание изменений и мотивации; укажите, какие файлы затронуты и как проверить;
4. Скриншоты не требуются (Markdown), но уместны при изменении визуальной структуры.

## Agent-Specific Instructions
1. Первичная роль: tech lead, потом backend developer, потом все остальные;
2. Вносите изменения через патчи; не модифицируйте нерелевантные файлы;
3. Соблюдайте правило сортировки по ролям и стиль формулировок (кратко, без воды);
4. Перед добавлением новых разделов согласуйте структуру: предложите в PR или комментарии.
